# 0711_plan1：统一 Media Item、GT-first Compare 与显式空间对齐

## 沟通记录与已确认决策

用户原始要求：

1. Pred 来源不应包含 Compare Run 的派生内容；Compare 已经是对已有视频进行比较，其结果不应再次进入后续 Compare 或盲评。
2. GT 需要按文件夹或 Collection 分组，例如先选 `test4k`，再筛选该组视频；只有同一 GT 及其对应 Pred 才能组合。
3. 推理和可视化通常不是原分辨率，因此严格对齐必须允许显式上/下采样，并向用户说明实际变换。

本轮问答锁定的产品决策：

| 议题 | 最终决定 |
|---|---|
| Compare 产物复用 | Compare Run 不再生成可复用 `pred_video`，Compare 结果永远不进入后续 Compare 或盲评来源。 |
| Compare 输入数量 | 一份 GT，最多两份对应 Pred。 |
| GT 选择 | 先选文件夹/Collection，例如 `test4k`，再筛选其中的视频。 |
| GT 身份 | 必须属于同一个 canonical GT asset；不按文件名或 SHA 跨文件夹合并。 |
| 总体架构 | 重建统一 Media Item 层。 |
| Item 边界 | `media_items` 管语义视频，`media_assets` 保留为物理文件，Member 连接两者。 |
| Campaign 流程 | GT 分组优先，再选视频和覆盖这些视频的方法 A/B。 |
| 时间对齐 | Item、帧映射、FPS、时间戳继续严格验证。 |
| 空间对齐 | 单 Pred 使用自身尺寸；双 Pred 使用像素面积较小者的完整尺寸。 |
| 宽高比变化 | VFIEval 已知 resize 允许拉伸，但必须记录；外部素材需显式确认。 |
| 历史数据 | 同步现有 GT Item，但不回填旧 Run Pred；只有升级后新产生且绑定 Item 的 Pred 可选择。 |
| 来源 Run 删除 | Compare 平时只引用原 Pred；来源删除前再冻结不可复用快照，冻结失败则不删除来源。 |

## 1. 统一 Media Item 身份层

新增：

- `media_items`
  - `id`、`collection_id`、`item_key`、`canonical_gt_asset_id`
  - `display_name`、`media_kind`、`state`、`metadata_json`
  - 创建、更新、删除时间
  - canonical GT 按精确 asset/source key 隔离。
- `media_item_members`
  - `item_id`、`asset_id`
  - `member_role`：`canonical_gt/model_pred/external_pred/compare_snapshot/evaluation_gt/evaluation_pred`
  - `producer_kind`、`producer_run_id`、`method_key`
  - `reusable_as_pred`
  - `temporal_mapping_json`、`spatial_origin_json`、`state`、时间字段。
- `run_media_item_bindings`
  - 连接 Run、Item 与 Member。
  - 区分 `source/pred_output/compare_gt/compare_pred`。
  - Compare binding 同时保留原始 Member 与当前有效 Member，便于删除前切换到快照。

稳定规则：

- 只有 `model_inference` 新 Run 产生的 `model_pred`，以及高级入口显式绑定的 `external_pred`，才能设置 `reusable_as_pred=true`。
- Compare 快照、Compare 派生产物和 Campaign 冻结资产一律不可复用。
- 新推理 Run 创建时绑定 source Item；发布 Pred 时写入对应 Item Member、`source_frame_indices` 和空间来源信息。
- 不按视频 stem、模型标签或 Compare track 名猜测身份。

## 2. GT-first 来源 API 与界面

新增主 API：

- `GET /api/media/item-groups?role=gt`
- `GET /api/media/items?group_id=...&q=...&page=...`
- `GET /api/media/items/{item_id}/predictions`
- `GET /api/media/methods?item_id=...`（支持重复 `item_id`）
- `GET /api/media/unbound-predictions`

Compare UI：

1. 选择 GT 分组。
2. 搜索并选择一个 GT Item。
3. 只展示该 Item 的 Pred。
4. 允许选择一至两份 Pred。
5. 显示方法、Run、原尺寸、帧映射和规范化摘要。
6. 不再提供无 GT 对比或跨 Item 组合。

Campaign UI：

1. 基本信息。
2. 选择一个 GT 分组。
3. 勾选多个 GT Item。
4. 查看方法覆盖率并选择 A/B。
5. 查看逐 Item 对齐矩阵。
6. 发布冻结包。

方法缺少某个 Item 时，该 Item 不可发布；用户可以取消勾选，不能静默跳过。

## 3. Compare Run 不再复制 Pred

新版请求：

```json
{
  "run_type": "video_compare",
  "media_item_id": 12,
  "pred_member_ids": [81, 93],
  "spatial_policy": {
    "mode": "smallest_pred",
    "filter": "lanczos",
    "allow_known_aspect_stretch": true
  },
  "metrics": ["vmaf"]
}
```

服务端必须验证：

- Pred Member 属于同一个 Item。
- 数量为 1–2。
- Member 可复用且来自有效模型 Run 或显式 External binding。
- owning Run 不是 `video_compare`。
- 时间映射完全兼容。

Compare Run：

- 保存输入 Item/Member binding、对齐计划、Diff、指标和报告。
- 不生成 `pred_video` Run artifact。
- 原 Pred 和 GT 通过 binding 播放。
- aligned 版本按需从 Compare cache 生成；cache 被回收后可从原输入重新生成。
- Diff 和指标必须使用实际规范化后的帧，不能依赖浏览器 CSS 缩放。

新增详情接口：

- `GET /api/runs/{id}/compare-inputs`
- `GET /api/runs/{id}/compare-inputs/{slot}/media?variant=original|aligned`

来源 Run 删除时：

1. 查找所有依赖该 Pred Member 的 Compare Run。
2. 优先硬链接/文件克隆，复制兜底，将快照保存到对应 Compare Run 的受控输入目录。
3. 注册 `compare_snapshot` Member，`reusable_as_pred=false`。
4. 原子切换 Compare binding。
5. 所有快照成功后才允许来源 Run 清理；任一失败则整个删除请求失败并可重试。
6. 删除 Compare Run 时一并删除其快照。

## 4. 时间严格、空间规范化

共享 `AlignmentPlan` 供 Compare 与 Campaign 使用。

时间验证：

- canonical Item 必须一致。
- `source_frame_indices` 长度正确、无越界。
- 双 Pred 必须拥有完全相同的有序帧映射。
- FPS 与可用时间戳必须匹配。
- 没有可信映射的素材只能接受原有 exact 时间对齐。
- resize 不能修复错视频、错帧或不同 GT。

空间目标：

- 一个 Pred：采用该 Pred 宽高。
- 两个 Pred：采用像素面积较小的 Pred 完整宽高。
- 面积相同时依次比较最大边、宽、高，确保结果确定。
- GT 和另一份 Pred 使用 LANCZOS 规范化到目标尺寸。
- VFIEval 已记录的非等比 resize 可拉伸。
- External Pred 宽高比变化必须在高级入口显式确认。

对齐报告必须记录原始/目标宽高、`scale_x/scale_y`、`none/upscale/downscale/mixed`、`aspect_changed`、插值方法、时间映射摘要和 Alignment Plan 指纹。该指纹必须进入 Compare cache、指标 cache、Campaign manifest 和分析详情。

## 5. Campaign V2 接入 Item

Preview 请求：

```json
{
  "media_item_ids": [12, 15, 18],
  "method_a": {"kind": "run", "run_id": 41},
  "method_b": {"kind": "run", "run_id": 52},
  "spatial_policy": {"mode": "smallest_pred"}
}
```

数据库调整：

- `evaluation_items_v2` 增加 `media_item_id`。
- `evaluation_bindings_v2` 增加 source/frozen Member 外键。
- 旧 asset 字段保留兼容读取。

发布阶段：

- 每个 Item 独立生成 Alignment Plan。
- staging 中物化规范化 GT/A/B。
- 生成 Diff、SHA-256 manifest 和完整变换报告。
- 全部成功后原子发布。
- 冻结 Member 标记不可复用。
- 已发布 Campaign 继续不依赖来源 Run。
- 人工分析和客观指标继续分离。

## 迁移与兼容

- 升级前使用现有 SQLite 备份机制。
- Catalog 同步为当前 folder/upload GT 建立 Item。
- 不为升级前旧 Run Pred 创建 Item Member。
- 旧 Run、旧 Compare 和已发布 Campaign 保持可查看。
- 旧 Compare 产物从所有新版来源选择器隐藏。
- 未发布且缺少 Item binding 的旧 Campaign Draft 不允许继续发布，应提示重新创建。
- `/api/media/sources`、`/api/media/run-outputs` 和 `/api/compare-sources/*` 保留为兼容 wrapper，但只返回新版已绑定 Member。
- 旧 `reference/distorted` payload 仅兼容解析；最终必须解析到同一个 Item，否则拒绝。
- 不自动删除历史数据。

## 实施顺序

1. Item 基础层：数据表、索引、Catalog 同步、Run binding、审计接口。
2. Alignment 与 Compare：共享 Planner/Materializer、Item payload、cache、详情、停止生成 `pred_video`、删除前快照。
3. GT-first UI 与 Campaign：两级 GT 选择、方法覆盖矩阵、规范化冻结、文档与完整验收。

## 测试与验收

- `test4k` 等文件夹先选组、再选视频，搜索只作用于当前组。
- 不同文件夹同名或 SHA 相同的 GT 不共享 Pred。
- 跨 Item Pred 的前端/API 请求均失败。
- Compare 只能选择 1–2 个 Pred，且不产生 `pred_video`。
- Compare/旧未绑定 Run 不进入来源；cache 清理后 aligned 输入可重建。
- 来源删除前快照成功才允许清理；失败可重试且来源仍保留。
- 覆盖单/双 Pred、上采样、下采样、mixed、宽高比拉伸及对齐报告。
- Item、帧索引、FPS 或时间戳不一致继续失败。
- Diff、VMAF 等指标读取规范化媒体且 cache key 含 Alignment Plan。
- Campaign 覆盖矩阵、原子冻结、Run 删除后播放及人工/客观分析分离。
- 旧 Pred 不回填、不复活；已发布旧 Campaign 和历史 Compare 可读。
- 每阶段运行 `python -m unittest discover -s tests`、三份 JavaScript 语法检查、`git diff --check` 和浏览器端到端验收。

## 明确默认与边界

- 一个 `media_item` 对应一份精确 canonical GT asset。
- Campaign 一次只选择一个 GT 分组，但可包含多个 Item。
- Compare 最多两份 Pred；Campaign 固定两种方法。
- 不进行旧 Pred 自动或人工猜测回填。
- External Pred 仅在高级入口显式绑定 Item。
- Compare 不主动复制 Pred；仅在来源删除前创建不可复用保护快照。
- 原始媒体永不覆盖。
- 不新增训练、PSNR、远程 Worker、对象存储、账号权限或新指标。
- 后续同类沟通文档继续采用 `plans/MMDD_planN.md`：先记录原始需求和问答决策，再在同一文件中维护最终执行计划。
